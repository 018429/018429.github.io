操作系统

多级反馈队列

本章将介绍一种著名的调度方法——多级反馈队列（Multi-level Feedback Queue, MLFQ）。1962年，Corbato首次提出多级反馈队列，应用于兼容分时共享系统（CTSS）。

多级反馈队列需要解决两方面的问题。首先，它要优化周转时间，这可以通过先执行短工作来实现。然而，操作系统通常不知道工作要运行多久，而这又是SJF（或STCF）等算法所必须的。其次，MLFQ希望给交互用户（如用户坐在屏幕前，等着进程结束）很好的交互体验，因此需要降低响应时间。然而，像轮转（RR）这样的算法虽然降低了响应时间，周转时间却很差。所以这里的问题是：通常我们对进程一无所知，应该如何构建调度程序来实现这些目标？调度程序如何在运行过程中学习进程的特征，从而做出更好的调度决策？

关键问题：没有完备的知识如何调度

没有工作长度的先验知识，如何设计一个能同时响应时间和周转时间的调度程序？

提示：从历史中学习

多级反馈队列是利用历史经验预测未来的一个典型的例子，操作系统中有很多地方采用了这种技术（同样存在于计算机科学领域的很多其他地方，比如硬件的分支预测及缓存算法）。如果工作有明显的阶段性行为，因此可以预测，那么这种方式会很有效。

1. MLFQ: 基本原则

MLFQ中有许多独立的队列（queue），每个队列有不同的优先级（priority level）。任何时刻，一个工作只能存在一个队列中。MLFQ总是优先执行较高优先级的工作（即在较高级队列中的工作）。

当然，每个队列中可能会有多个工作，因此具有同样的优先级。在这种情况下，我们就对这些工作采用轮转调度。

因此，MLFQ的调度策略的关键在于如何设置优先级。MLFQ没有为每个工作指定不变的优先级，而是根据观察到的行为调整它的优先级。例如，如果一个工作不断放弃CPU去等待键盘输入，这是交互型进程的可能行为，MLFQ因此会让它保持较高优先级。相反，如果一个工作长时间地占用CPU，MLFQ会降低其优先级。通过这种方式，MLFQ在进程运行过程中学习其行为，从而利用工作的历史来预测它未来的行为。

至此，我们得到了MLFQ的两条基本规则。

  • 规则1：如果A的优先级 > B的优先级，运行A（不运行B)。
  • 规则2: 如果A的优先级 = B的优先级，轮转运行A和B。

如果要在某个特定时刻展示队列，可能会看到如下内容（见图1）。图1中，最高优先级有两个工作（A和B），工作C位于中等优先级，而D的优先级最低。按刚才介绍的基本规则，由于A和B有最高优先级，调度程序将交题的调度它们，可怜的C和D永远没有机会运行，太气人了！

计算机网络

1. 网络基础知识

1.1. 协议分层与OSI参考模型

  • 每个分层都接收由它下一层所提供的特定服务，并且负责为自己的上一层提供特定的服务。上下层之间进行交互时所遵循的约定叫做接口。同一层之间交互所遵循的约定叫做”协议“
  • OSI参考模型分为7层，从上到下分别是：
    • 应用层：针对特定应用的协议
    • 表示层：设备固有数据格式和网络标准数据格式的转换
    • 会话层：通讯管理。负责建立和断开通信连接
    • 传输层：管理两个节点之间的数据传输。负责可靠传输
    • 网络层：地址管理与路由选择
    • 数据链路层：互联设备之间传送和识别数据帧
    • 物理层：以0、1 代表高低电压，灯光闪灭。界定连接器和网线的规格。

1.2. 基础知识

  • 时延带宽积
    • 时延带宽积又称为以比特为单位的链路长度
    • 即“某段链路现在有多少比特”容量
  • 往返时延RTT
    • 从发送方发送数据开始，到发送方接收到接收方的确认，总共经历的时延。
    • RTT越大，在收到确认之前，可以发送的数据越多。
    • RTT包括：往返传播时延=传播时延*2;末端处理时延
  • 利用率
    • 信道利用率：有数据通过的时间/（有+无)数据通过时间
    • 网络利用率：信道利用率加权平均值

4. IP协议

IP: Internet Protocol，网际协议。

4.1. IP 即网际协议

TCP/IP的心脏是互联网层。这一层主要由IP和ICMP(Internet Control Message Protocol)两个协议组成。

4.2. IP 基础知识

IP大致分为三大作用模块，它们是IP寻址、路由(最终节点为止的转发)，以及IP分包与组包。

数量链路的抽象化

  • IP是实习多个数据链路之间通信的协议。数据链路的地址可以抽象化为IP地址，因此对于IP的上一层来说，不论底层数据链路使用以太网还是LAN异或是PPP,都将被一视同仁。
  • 不同数据链路MTU不同。以太网是1500字节，FDDI是4352字节，ATM是9180字节。IP上一层可能要求传送比这些MTU更多的字节数据，为了解决这个问题，IP进行分片处理(IP Fragmentation)。所谓分片就是将较大的IP包分成许多较小的IP包。
  • IP面向无连接。为什么IP要采用面向无连接呢？
    • 一是为了简化，二是为了提速。面向连接比面向无连接处理相对复杂。每次通信之前都要建立连接又会降低处理速度。需要有连接时可以委托上一层提供此项服务。因此iP为了实现简单化和高速化采用了面向无连接的方式。
  • 为什么不让IP具有可靠传输的功能，从而把TCP/IP合并到一起？
    • 如果要一种协议规定所有的功能和作用，那么该协议的具体实施和编程会非常困难。按照网络分层后，针对每层进行编程更有利于该协议实现。分层也有利于后续对这些协议进行拓展和性能优化。

4.3. IP地址的基础知识

IP地址的定义

IP地址（IPv4地址）由32位正整数来表示。最大允许43亿台计算机连接网络。实际连接到网络计算机少很多。

IP地址由网络和主机两部分标识组成。

IP地址的分类

IP地址分为A类，B类，C类，D类。

  • A类地址
    • 首位以0开头。1到8位是网络标识。0.0.0.0~127.0.0.0是A类的网络地址。后24位相当于主机标识。一个网段可以容纳主机地址上限为16777214个。
  • B类地址
    • 前两位是01，第1位到第16位是网络标识。十进制表示128.0.0.1~`191.255.0.0`是B类的网络地址。一个网段可以容纳的主机地址上限为65534个。
  • C类地址
    • 前三位是110,第一位到第24位是网络标识。192.168.0.0~`239.255.255.0`是C类的网络地址。一个网段可以容纳的主机地址上限为254个。
  • D类地址
    • 前四位为1110,第1位到第32位是网络标识。224.0.0.0~`239.255.255.255`是D类的网络地址。D类地址没有主机标识，常被用于多播。
  • 注意事项
    • 用比特位表示主机地址时，不可以全部为0或1。全部为0表示对应的网络地址或IP地址不可获知才使用。全部为1的主机地址通常作为广播地址。这也是为什么C类地址每个网段最多有254(256 - 2)个主机的原因。

广播地址

  • 本地广播
  • 直接广播

IP多播

多播用于将包发送给组内的所有主机。由于其直接使用了IP协议，因此不存在可靠传输。

子网掩码】

引入子网以后，一个IP地址就有了两种识别码，一是IP地址本身，另一个是表示网络部的子网掩码。两种表示方式：

  • ip地址：172.20.100.52
  • 子网掩码：255.255.255.192
  • IP地址：172.20.100.52/26

1. 数据库管理软件的由来

1.1. 程序所有的组件不可能运行在一台机器上

因为单点故障和单台机器性能有限，于是通过水平拓展方式增强整体性能。

各个机器上的程序是一个整体，所有组件的数据需要共享，于是想到数据和程序分离，文件存放于一台机器，多台机器通过网络访问文件(socket实现)。

我们必须写一个socket服务端来管理这台机器，然后写一个socket客户端，完成如下功能：

  • 远程连接（支持并发）
  • 打开文件
  • 读写（加锁）
  • 关闭文件

2. 数据库概述

2.1. 什么是数据

描述事物的符号记录称为数据

2.2. 什么是数据库

存放数据的仓库，数据库中的数据按一定的数据模型组织，描述和存储，具有较小的冗余度，较高的数据独立性和易拓展性，并可为各种用户共享。

2.3. 什么是数据库管理系统

在了解了Data和DB之后，如何科学的组织和存储数据，如何高效获取和维护数据成为了关键。这就用到了系统软件——数据库管理系统如Mysql、Oracle

  • 数据库：文件夹，组织文件/表
  • 表：文件，存放多行内容/多条记录

3. 存储引擎

3.1. 什么是存储引擎

存储引擎说白了就是如何存储数据、如何为存储数据建立索引和如何更新、查询数据等技术的实现方法。因为在关系型数据库中数据的存储是以表的形式存储的，所以存储引擎也可以称为表类型（即存储和操作此表的类型）。在Oracle和SQL Server等数据库中只有一种存储引擎，而Mysql数据库提供了多种存储引擎。

3.2. MySQL支持的存储引擎

3.2.1. InnoDB存储引擎

支持事务，设计目标是面向联机事务处理(OLTP)的应用。其特点是行锁设计，支持外键，并支持类似Oracle的非锁定读，即默认读取操作不会产生锁。

InnoDB存储引擎通过多版本并发控制(MVCC)来获得高并发性，并且实现了SQL标准的四种隔离级别，默认为REPEATABLE级别，同时使用一种称为next-key locking的策略避免幻读。

对于表数据存储，InnoDB存储引擎采用了聚集(clustered)的方式，每张表按主键顺序进行存储，如果没有显式定义主键，存储引擎为每一行生成一个6字节的ROWID作为主键。

3.2.2. MyISAM存储引擎

不支持事物、表锁设计，支持全文索引，主要面向一些OLAP数据库应用。

3.3. 基于存储引擎的模型

大致基于存储引擎的模型分：B+ tree和LSM tree，复杂。

bitcask这种模型，本质上也算LSM tree的范畴，简单。

4. 事物

在传统的关系式数据库中，常常用ACID性质来检验事务功能的安全性。

4.1. A: 原子性

所谓原子性，一个事务(transaction)中的所有操作，或者全部完成，或者全部不完成，不会结束在中间某个环节。实现事务的原子性，要支持回滚操作，在某个操作失败后，回滚到事务执行之前的状态。一般的做法是类似数据快照的方案。

4.2. C: 一致性

在事务开始之前和事务结束之后，数据库的完整性没有被破化。这表示写入的数据必须完全符合预期的。

4.3. I: 隔离性

数据库允许多个并发事务同时对其数据进行读写和修改的能力，隔离性可以防止多个事务并发执行时由于交叉执行而导致的数据不一致。

隔离级别低到高：

4.3.1. 未提交读（READ UNCOMMITTED）

这个是最低的隔离级别。允许脏读(dirty reads)，事务可以看到其他事务尚未提交的修改。

4.3.2. 在提交读（READ COMMITTED）

定义：在这个隔离级别中，基于锁机制并发控制的DBMS需要需要对选定对象的写锁一直保持到事务结束，但是读锁在SELECT操作完成后马上释放（因此“不可重复读”现象可能会出现）。看下“不可重复读”的定义：在一次事务中，当一行数据获取两遍得到不同的结果表示发生了”不可重复读“。

4.3.3. 在可重复读（REPEATABLE READS）

定义：在这隔离级别中，基于锁机制并发控制的DBMS需要对选定对象的读锁(read locks)和写锁(write locks)一直保持到事务结束，但是不要求”范围锁“，因此可能出现”幻影读“。

关于幻影读的定义，指在事务执行过程中，当两个完全相同的查询语句执行得到不同的结果集。这种现象称为”幻影读(phantom read)“，有些人也叫它幻读。

4.3.4. 可串行化（Serializable）

定义：这个隔离级别是最高的。避免了所有上面的“脏读”，”不可重复读“，”幻影读“现象。

4.4. D: 持久化 durability

事务处理结束以后，对数据修改就是永久的，即使系统故障machine crashes也不会丢失。
